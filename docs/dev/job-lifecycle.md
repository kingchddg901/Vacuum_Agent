# Cleaning Job Lifecycle

This document traces every state, transition, side effect, and event across the full life of a room-cleaning job — from queue build through finalization. A developer should be able to follow the entire flow from source code using this document.

**Primary source files:**
- `custom_components/eufy_vacuum/core/manager.py`
- `custom_components/eufy_vacuum/__init__.py`
- `custom_components/eufy_vacuum/learning/job_finalizer.py`
- `custom_components/eufy_vacuum/learning/services.py`
- `custom_components/eufy_vacuum/jobs/job_monitor.py`
- `custom_components/eufy_vacuum/const.py`

## 1. Pre-job: Queue Build

### `build_queue`

`EufyVacuumManager.build_queue(vacuum_entity_id, map_id)` reads the managed room records from the map bucket and passes them to `build_queue_from_managed_rooms`. The result is written to `self.data["queue"][vacuum_entity_id][map_id]`. It also updates the runtime object: `runtime.selected_map_id` and `runtime.queue_room_ids`.

The queue contains only rooms whose `enabled` flag is `True`. The output shape is:
```
{
  "vacuum_entity_id": str,
  "map_id": str,
  "room_count": int,
  "queue_room_ids": list[int],
  "queue_rooms": list[dict],
}
```

### `build_room_payload`

`EufyVacuumManager.build_room_payload(vacuum_entity_id, map_id)` builds the `room_clean` command payload. Before passing rooms to the payload builder it applies carpet/mop invariants via `_protected_room_config` on every room. It fetches:
- the current queue state for `queue_room_ids`
- stored room profiles from `self.data["profiles"]["room_profiles"]`
- vacuum capabilities via `get_vacuum_capabilities`

The result is stored at `self.data["payloads"][vacuum_entity_id][map_id]` and contains the raw `payload` dict plus a `resolved_rooms` list with full per-room settings.

### `get_start_status` — blocker reasons

`get_start_status` calls `_build_effective_start_plan` (which evaluates all room rules against live HA states), `get_lifecycle_state`, and `get_onboarding_state` and assembles a status dict. It returns early with `blocked: True` and the following exact `reason` strings in priority order:

| `reason` | Condition |
|---|---|
| `"job_paused"` | `active_job["status"] == "paused"` |
| `"onboarding_required"` | One or more enabled rooms are missing a floor type |
| `"all_selected_rooms_blocked"` | All selected rooms were blocked by rules; none remain |
| `"no_target_map"` | `selected_map_id` is empty |
| `"map_mismatch"` | `selected_map_id` != vacuum's `active_map` sensor |
| `"no_rooms_selected"` | Queue is empty (`queue_count <= 0`) |
| `"invalid_payload"` | Payload room count is 0 |
| `"mid_job_service"` | Dock or task status is in a hard service state (washing, recycling waste water, emptying dust) |
| `"active_job_running"` | A room-clean job is already active |
| `"vacuum_busy"` | Vacuum is busy and not dockable/idle |
| `"incomplete_access_graph"` | Room access graph is partially configured |
| `"access_graph_required_for_rules"` | Rules are present but no access graph exists |
| `"access_graph_required"` | Blocker rules exist without an access graph |

The lifecycle state `"dock_drying"` is a non-blocking warning — it sets `blocked: False` with `warning: True` and `reason: "dock_drying"`.

#### Preflight rule evaluation (`_build_effective_start_plan`)

This is the authoritative rule-evaluation site for job start (the only other site is `get_runtime_path_block_report` for mid-job path changes). At the moment `get_start_status` is called:

1. All managed rooms with automation rules are loaded.
2. Blocker rules are evaluated against live HA entity states. Matching rooms go into `direct_blocked`.
3. Modifier rules are evaluated; matching changes are accumulated in `modifier_matches`.
4. Access-graph propagation: rooms that require access through a directly-blocked room are also marked blocked.
5. `included_room_ids` = selected minus blocked.
6. `requires_confirmation` becomes `True` when `blocked_ratio_time >= 0.20` or `blocked_ratio_rooms >= 0.40`. When confirmation is required, a `confirm_token` (opaque hash of the preflight parameters) is generated.

## 2. Job Start

### `start_selected_rooms`

`async EufyVacuumManager.start_selected_rooms(vacuum_entity_id, map_id, confirm_reduced_run, confirm_token, path_block_action, pause_timeout_minutes_override)`

**Step-by-step flow:**

1. **Blocker check** — calls `get_start_status`. If `blocked`, returns immediately with `started: False`.
2. **Confirmation handshake** — if `requires_confirmation` is set, start is blocked unless:
   - `confirm_reduced_run=True` (bypass flag for automations), **or**
   - the caller supplies a `confirm_token` that matches the one generated by the preflight
   If neither condition is met, returns `{"started": False, "reason": "confirmation_required", "confirm_token": <token>}`.
3. **Rebuild plan** — calls `_build_effective_start_plan` again (rules may have changed since `get_start_status` was called) and writes the final queue/payload into storage.
4. **Vacuum entity check** — if the HA state object is missing, returns `{"started": False, "reason": "vacuum_missing"}`.
5. **Command dispatch** — calls `vacuum.send_command` with `command="room_clean"` and the payload as `params` (blocking call).
6. **`active_job` initialisation** — calls `build_active_job_state` then enriches the dict with:
   - `job_metadata` from `build_job_metadata_from_payload`
   - `job_id` (generated UUID-like key)
   - `started_at` (UTC ISO timestamp)
   - `battery_start`
   - `current_room_started_at` = `started_at`
   - `path_block_action` (normalised; defaults to `"event_only"`)
   - `pause_timeout_minutes` (from config or override)
   - `water_estimate` from `get_planned_job_estimate`
7. **Storage write** — saved to `self.data["active_jobs"][vacuum_entity_id][map_id]`.
8. **Room-started event** — if `current_room_id` is set, fires `eufy_vacuum_room_started` (see event table below).
9. **Runtime update** — `runtime.active_job_room_ids` is set; room selections are cleared via `_clear_room_selections_after_start`.
10. **Learning snapshot** — `save_learning_snapshot_for_active_job` is called. The snapshot freezes the queue/payload/active_job state to disk non-blockingly (via executor) as the source of truth for finalization. Any failure is caught and logged; it does not abort the job.

### Default `active_job` state fields

```python
{
  "vacuum_entity_id": str,
  "map_id": str,
  "queue_room_ids": [],
  "queue_stable_keys": [],
  "queue_rooms": [],
  "payload": {"map_id": str, "rooms": []},
  "resolved_rooms": [],
  "room_count": 0,
  "status": "idle",           # → "started" after start
  "paused_at": None,
  "paused_duration_seconds": 0,
  "completed_room_ids": [],
  "completed_rooms": [],
  "current_room_id": None,
  "current_room_started_at": None,
  "current_room_paused_seconds": 0,
  "observed_mid_job_recharge": False,
  "observed_mid_job_recharge_started_at": None,
  "observed_mid_job_recharge_count": 0,
  "recharge_seconds_accumulated": 0,
  "pending_mid_job_recharge_return": False,
  "pending_mid_job_recharge_return_at": None,
  "observed_mop_wash_count": 0,
  "observed_mop_wash_last_at": None,
  "state_transitions": [],
  "water_estimate": None,
  "path_block_action": "event_only",
  "pause_timeout_minutes": 0,
  "has_observed_active_lifecycle": False,
}
```

## 3. Active Job Monitoring

### `get_job_progress_snapshot`

`EufyVacuumManager.get_job_progress_snapshot(vacuum_entity_id, map_id)` is the main polling endpoint called by the card. It does not mutate storage unless a timing rollover occurs (see below). Each call:

1. Loads and normalises `active_job` via `_normalize_active_job`.
2. Calls `get_lifecycle_state` for the current lifecycle context.
3. Reads the current battery level.
4. **Timeline construction** — if the learning system is available and `resolved_rooms` is set:
   - First call: `learning.estimate_from_manager` produces a full `room_timeline` (`timeline_source = "estimate"`).
   - After any room completes: `learning.reanchor_timeline` replaces the estimate using actual completed-room durations (`timeline_source = "reanchored"`).
   - No learning system: `timeline_source = "none"`.
5. **Current room resolution** — reads `active_job["current_room_id"]`. If that ID is not in `unresolved_room_ids`, falls back to the first unresolved room.
6. **Elapsed time** — calls `_compute_current_room_elapsed_minutes`, which computes wall-clock elapsed since `current_room_started_at`, then subtracts accumulated `current_room_paused_seconds` plus any ongoing pause.
7. **Timing rollover** — calls `_maybe_roll_current_room_by_timing`.
8. **Bounds-exit detection** (`awaiting_bounds_exit`).
9. **Stall detection**.

### `awaiting_bounds_exit` logic

After the timing rollover attempt, if the room did **not** roll (i.e., `current_room_id` is unchanged), the snapshot checks whether elapsed time has passed the timing completion threshold. If it has, `awaiting_bounds_exit` is set to `True`. This signals the card to switch to a short-interval poll (~5 s) because the robot is still physically inside the room and the rollover gate is blocked.

### Current room determination

`_derive_active_job_current_room_id(active_job)` returns the first room ID from `resolved_rooms` (or from `queue_room_ids` as fallback) that is not already in `completed_room_ids`. This becomes `current_room_id` after each room completes.

### Stall detection

A stall is detected when:
- `awaiting_bounds_exit` is already `True` (timing threshold exceeded), **and**
- `current_room_elapsed_minutes >= threshold * 2.0`

The constant `_STALL_RATIO = 2.0` means elapsed must reach **2× the timing completion threshold** (see threshold formula in section 4).

The event `eufy_vacuum_stall_detected` fires at most once per room per job. Rooms that have already received a stall notification are tracked in `active_job["_stall_notified_room_ids"]` (a list of room IDs written back to storage). On subsequent snapshot calls the set is checked and the event is suppressed.

## 4. Room Transitions

### What triggers `room_started` / `room_finished`

`eufy_vacuum_room_started` fires in two situations:
- At job start (`source: "job_start"`), if `current_room_id` is non-null when `active_job` is first written.
- After each timing rollover (`source: "timing_rollover"`), for the **next** room.

`eufy_vacuum_room_finished` fires from `_maybe_roll_current_room_by_timing` after a room is rolled over (`source: "timing_rollover"`).

There is no event-driven (HA entity state) room transition mechanism. All transitions are timing-based.

### Timing rollover mechanism (`_maybe_roll_current_room_by_timing`)

Called from `get_job_progress_snapshot`. Steps:

1. Requires `active_job["status"] == "started"` and a valid `current_room_id`.
2. Checks `current_room_elapsed_minutes >= _timing_completion_threshold_minutes(current_room)`.
3. If timing is met, checks whether the robot has physically left the room bounds (via `mapping_manager.get_room_bounds_snapshot`). If room bounds exist and the robot is still inside, rollover is blocked — `awaiting_bounds_exit` will be `True` on the next snapshot call.
4. If bounds are unavailable (room not yet mapped), timing alone determines rollover.
5. On rollover: calls `record_completed_room` (which advances `current_room_id`, resets `current_room_started_at` and `current_room_paused_seconds`), then fires `eufy_vacuum_room_finished` and `eufy_vacuum_room_started`.
6. Persists state asynchronously via `_async_save_logged`.

### Timing completion threshold formula

`_timing_completion_threshold_minutes(room)`:

```
threshold = estimated_minutes + slack_minutes

where:
  overrun_ratio = 0.06   if confidence_score >= 0.85
                  0.10   if confidence_score >= 0.65
                  0.15   if confidence_score >= 0.45
                  0.22   otherwise

  slack_minutes = max(0.75, estimated_minutes * overrun_ratio)
  + 1.0 if sample_count <= 1
  + 0.5 if sample_count <= 3
  + min(estimated_minutes * drift_ratio * 0.25, 1.5) if drift_ratio > 0

  slack_minutes is capped at max(4.0, estimated_minutes * 0.35)
```

The stall threshold is `2 × _timing_completion_threshold_minutes`.

### `reanchor_learning_timeline`

`learning.reanchor_timeline(original_estimate, completed_rooms, reanchor_at, current_battery, ...)` takes the original pre-job estimate and a list of `{room_id or slug, actual_duration_minutes}` entries for completed rooms. It replaces estimated durations for those rooms with actuals, then recalculates all downstream ETAs and battery projections from the reanchor point. Called from `get_job_progress_snapshot` on every snapshot call once any room has completed.

## 5. Job End Paths

There are five ways a job ends:

### 5a. Normal completion

**Trigger:** The lifecycle listener in `__init__.py` observes `task_status == "completed"` AND `active_cleaning_target` is cleared (in `{"", "unknown", "unavailable", "none", "null"}`), while `active_job["has_observed_active_lifecycle"] == True`.

**Code path:** `_handle_lifecycle_change → _process()` in `_register_lifecycle_listeners`. The guard `has_observed_active_lifecycle` prevents a stale pre-run dock state (e.g., `dock_drying`) from completing the job before it actually started. When completion signals are met, `finalize_learning_for_active_job` is called, then `mark_active_job_finalized`, then `EVENT_JOB_FINISHED` is fired.

### 5b. Manual cancel (service call)

**Trigger:** `eufy_vacuum.cancel_active_job` service is called.

**Code path:** `async_cancel_active_job` → calls `vacuum.return_to_base` (blocking), then polls every 2 s for up to 30 s (`_CANCEL_CONFIRM_TIMEOUT_S`) waiting for `vacuum_state in {"docked", "idle"}` or `task_status in {"completed", "complete"}`. If the device does not confirm within 30 s, finalization proceeds anyway (logged as a warning). Calls `finalize_learning_for_active_job` with `forced_outcome_status="cancelled"` and `forced_lifecycle_state="job_cancelled"`, then `mark_active_job_finalized`. The caller fires `EVENT_JOB_FINISHED`.

### 5c. Pause timeout

**Trigger:** A paused job whose `pause_timeout_minutes > 0` has been paused longer than the configured limit.

**Code path:** `_register_pause_timeout_listener` sets a 1-minute `async_track_time_interval` tick. On each tick, `get_paused_job_timeout_report` is called for each known job. If it returns a report, `async_cancel_active_job` is called with `forced_lifecycle_state="pause_timeout_cancelled"` and `cancel_reason="pause_timeout"`. `EVENT_JOB_FINISHED` is fired by the tick handler.

### 5d. Path blocker cancel

**Trigger:** A watched HA entity whose state change matches a `blocker` rule fires while a job is active, and `path_block_action` is `"cancel_and_event"`.

**Code path:** `_register_path_blocker_listeners` watches all entities referenced by blocker rules. On state change: `get_runtime_path_block_report` re-evaluates the rules. Based on `path_block_action`:
- `"event_only"` — fires `eufy_vacuum_path_blocked` only.
- `"pause_and_event"` — calls `async_pause_active_job`, then fires `eufy_vacuum_path_blocked`.
- `"cancel_and_event"` — calls `async_cancel_active_job`, fires `EVENT_JOB_FINISHED`, then fires `eufy_vacuum_path_blocked`.

### 5e. Vacuum returning-to-base detection (early return / cancel heuristic)

**Trigger:** During finalization, `_detect_cancel_likely_run` is called when `forced_outcome_status` is `None` (i.e., the outcome was not explicitly forced). It examines the job's `state_transitions` log.

**Conditions for `cancel_likely=True`:**
1. Single-room job only.
2. Transition history must contain `cleaning → returning` (direct, via `task_status`) **or** `paused → returning` pattern.
3. No "stronger service state" in the transition history (e.g., `"returning to charge"`, `"washing mop"`, `"emptying dust"`).
4. `actual_cleaning_minutes < 1.5` (absolute floor), **or** `actual_cleaning_minutes < expected_room_minutes * 0.4` (relative threshold).

When detected, `outcome_status` is overridden to `"cancelled"` with `lifecycle_name = "cancel_likely"`.

## 6. Finalization

### `finalize_learning_for_active_job` (manager entry point)

`async EufyVacuumManager.finalize_learning_for_active_job(...)` reads `started_at` and `battery_start` from the active job, reads current battery as `battery_end`, then delegates to `learning.async_finalize_completed_job`. After the result returns, it calls `_ingest_completed_job_into_room_history` and fires the room-history-updated notification if anything was ingested.

### `finalize_from_manager_state` / `finalize_from_inputs` (LearningJobFinalizer)

The finalizer separates event-loop work from file I/O:

1. **`_collect_finalization_inputs`** (event loop): loads the in-memory live snapshot (falls back to disk), reads current `queue_state`, `payload_state`, `active_job_state`, and `lifecycle_state` from the manager. Determines `outcome_status` from lifecycle name or cancel detection heuristic. Reads `cleaning_time_seconds` from `sensor.<id>_cleaning_time` and `cleaning_area_m2` from `sensor.<id>_cleaning_area`. Returns a frozen inputs dict.

2. **`finalize_from_inputs`** (executor thread): pure computation and file I/O:
   - Builds a `completed_job` payload via `store.build_completed_job_payload`.
   - Calls `_apply_snapshot_estimates_to_completed_job` to attach pre-run estimated minutes, battery, and confidence onto each resolved room.
   - Calls `_apply_water_actuals` to compute actual water-used breakdown from end-of-run station water percent.
   - Writes `learning_context` (queue shape key, estimate delta, access graph metadata).
   - Saves the completed job JSON via `store.save_completed_job`.
   - Calls `_write_incomplete_run_log` (see section 7).
   - Calls `_update_trouble_rooms_log`.
   - Optionally rebuilds learned stats via `rebuilder.rebuild_all`.
   - Optionally derives room boundary data.

### Learning eligibility check

A job is **not** used for learning if `outcome_status` is any of:
- `"cancelled"` (`was_cancelled = True`)
- `"failed"` (`was_failed = True`)
- `"interrupted"` (`was_interrupted = True`)
- `is_test_job = True` (always `False` from the auto-finalization path)

Normal completions (`outcome_status = "completed"`) are eligible. The `used_for_learning` flag is passed through to the completed job's `outcome` dict and propagated to the `EVENT_JOB_FINISHED` payload.

### What gets written where

| Data | File / location |
|---|---|
| Live snapshot (job start) | `<config>/eufy_vacuum/learning/<vacuum_id>/live_snapshot.json` |
| Completed job record | `<config>/eufy_vacuum/learning/<vacuum_id>/jobs/<job_id>.json` |
| Incomplete run log | `<config>/eufy_vacuum/learning/<vacuum_id>/incomplete_run.json` |
| Trouble rooms log | `<config>/eufy_vacuum/learning/<vacuum_id>/trouble_rooms.json` |
| Rebuilt stats | `<config>/eufy_vacuum/learning/<vacuum_id>/learned_stats.json` |

## 7. Incomplete Run

### When `EVENT_RUN_INCOMPLETE` fires

`EVENT_RUN_INCOMPLETE` (`eufy_vacuum_run_incomplete`) is fired by the `handle_finalize_learning_job` service handler in `learning/services.py` after finalization completes, **if** `incomplete_run_log` is non-null and contains at least one `missed_room_id`.

The incomplete run log is only written when `outcome_status in {"cancelled", "failed", "interrupted"}`. Normal completions clear any stale log.

### `incomplete_run.json` contents

```json
{
  "schema_version": 1,
  "record_type": "incomplete_run_log",
  "vacuum_entity_id": "vacuum.alfred",
  "job_id": "job_2026-04-30T10-00-00",
  "map_id": "6",
  "outcome_status": "cancelled",
  "ended_at": "<ISO timestamp>",
  "queued_room_ids": [1, 2, 3, 4],
  "completed_room_ids": [1, 2],
  "missed_room_ids": [3, 4],
  "missed_rooms": [
    {"room_id": 3, "name": "Office"},
    {"room_id": 4, "name": "Bedroom"}
  ],
  "logged_at": "<ISO timestamp>"
}
```

### `retry_missed_rooms` flow

`handle_retry_missed_rooms` in `learning/services.py`:

1. Reads the incomplete run log from disk via `learning.get_incomplete_run_log`. Returns `{"started": False, "reason": "no_missed_rooms"}` if empty.
2. Resolves `map_id` from the log (overridable via call data).
3. Calls `core_manager.set_rooms_enabled_subset` to enable only the missed rooms and disable all others.
4. Calls `core_manager.build_queue` to rebuild the queue with those rooms.
5. Calls `core_manager.start_selected_rooms` (with `confirm_reduced_run` defaulting to `True` so automations bypass confirmation).
6. Persists state via `async_save`.
7. If started successfully, clears the incomplete run log via `learning.clear_incomplete_run_log`.
8. Returns the `start_selected_rooms` result plus `missed_room_ids` and `map_id`.

## 8. State Cleanup

After finalization, `mark_active_job_finalized` updates the active job record in-place rather than clearing it:

```python
active_job["status"] = "completed"
active_job["finalized"] = True
active_job["paused_at"] = None
active_job["has_observed_active_lifecycle"] = False
active_job["finalized_at"] = <from completed_job>
active_job["finalize_summary"] = {
    "job_id", "job_path", "used_for_learning",
    "sanity_passed", "sanity_flags", "learning_blockers", "status"
}
```

`runtime.active_job_room_ids` is reset to `[]`.

The queue state (`self.data["queue"]`) and payload state (`self.data["payloads"]`) are **not** automatically cleared after finalization. They persist until the next `build_queue` / `build_room_payload` call (e.g., when the user selects rooms for the next job). The active job record itself remains at `status: "completed"` until explicitly cleared via `clear_active_job` or overwritten by the next job start.

## 9. Event Timeline

Chronological table of every HA event fired during a complete job (single successful run, no cancellations or stalls).

| # | Event | When fired | Key payload fields |
|---|---|---|---|
| 1 | `eufy_vacuum_room_started` | Immediately after `vacuum.send_command` succeeds, if `current_room_id` is non-null | `vacuum_entity_id`, `map_id`, `job_id`, `room_id`, `room_name`, `started_at`, `source: "job_start"`, `completed_room_ids: []` |
| 2 | `eufy_vacuum_room_finished` | Each time timing rollover fires (room N complete) | `vacuum_entity_id`, `map_id`, `job_id`, `room_id`, `room_name`, `completed_at`, `source: "timing_rollover"`, `actual_duration_minutes`, `confidence`, `completed_room_ids` |
| 3 | `eufy_vacuum_room_started` | Immediately after each `room_finished`, for the next room | `vacuum_entity_id`, `map_id`, `job_id`, `room_id`, `room_name`, `started_at`, `source: "timing_rollover"`, `completed_room_ids` |
| 4 | `eufy_vacuum_stall_detected` | Once per room per job, when elapsed >= 2× timing threshold and bounds haven't cleared | `vacuum_entity_id`, `map_id`, `room_id`, `room_name`, `elapsed_minutes`, `expected_minutes`, `stall_ratio` |
| 5 | `eufy_vacuum_path_blocked` | When a blocker entity changes state during a job (any `path_block_action`) | `vacuum_entity_id`, `map_id`, `path_block_action`, `action_taken`, `affected_remaining_room_ids` |
| 6 | `eufy_vacuum_job_finished` | After finalization completes (normal, cancel, timeout, or path-blocker cancel) | `vacuum_entity_id`, `map_id`, `job_id`, `status`, `reason_detail`, `used_for_learning`, `finalized_at`, `room_count`, `job_path` |
| 7 | `eufy_vacuum_run_incomplete` | After `job_finished`, only when `outcome_status` is cancelled/failed/interrupted and rooms were missed | `vacuum_entity_id`, `job_id`, `outcome_status`, `missed_room_ids` (list of int), `missed_rooms` (list of `{room_id, name}`) |

**Notes:**
- Events 2 and 3 repeat for each room beyond the first. A 4-room job produces 3 pairs of room_finished/room_started plus the initial room_started.
- Event 4 may fire for any room that stalls; it fires at most once per room even if the snapshot is polled many more times.
- Events 5, 7 are conditional and may not appear in every job.
- `eufy_vacuum_path_blocked` and `eufy_vacuum_job_finished` can both fire in the same job if `path_block_action == "cancel_and_event"`.
